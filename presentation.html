Problem — co chcemy rozwiązać?

Każda zmiana w CSS, szablonie lub bibliotece ikon może niezauważalnie zmienić wygląd komponentu.

  • Code review nie wychwytuje zmian wizualnych
  • Testy jednostkowe / e2e sprawdzają zachowanie, nie wygląd
  • Ręczne przeglądanie każdej story po każdym PR jest nieefektywne
  • Regresje wykrywane dopiero na produkcji

Przykładowe regresje

  • Zmiana padding w tokenie design systemu
  • Upgrade biblioteki ikon → inny rozmiar SVG
  • Nadpisanie font-family przez globalny styl
  • Zmiana wersji Angular Material
  • Refactor SCSS → niezamierzona zmiana specificity

Podejście 1 — kod testu

visual-tests/storycap.test.ts

import { configureToMatchImageSnapshot }
  from 'jest-image-snapshot';
import * as fs from 'fs';
import * as path from 'path';

const toMatchImageSnapshot =
  configureToMatchImageSnapshot({
    customSnapshotsDir:
      '__screenshots_baseline__',
    failureThreshold: 0.02,        // 2%
    failureThresholdType: 'percent',
  });

expect.extend({ toMatchImageSnapshot });

describe('Button component', () => {
  it('Primary button matches baseline', () => {
    const png = fs.readFileSync(
      '__screenshots__/Components_Button_Primary.png'
    );
    (expect(png) as any)
      .toMatchImageSnapshot({
        customSnapshotIdentifier:
          'button-primary',
      });
  });
});

Podejście 1 — zalety i wady

✅ Zalety

  • Screenshoty wszystkich stories automatycznie — nie trzeba pisać testu dla każdej
  • Doskonała integracja z istniejącymi pipeline'ami Jest
  • Prosta konfiguracja progu tolerancji (failureThreshold)
  • Storycap równolegle capture'uje wiele stories
  • Diff image jest czytelny i łatwy do debugowania
  • Znajomy ekosystem dla team'u używającego Jest

⚠️ Wady

  • Dwuetapowy pipeline — capture + compare są rozdzielone
  • Storycap oficjalnie wspiera Storybook 7/8 (używamy 10 z --legacy-peer-deps)
  • Screenshoty całej strony, nie tylko komponentu — tło i padding iFrame mogą wpływać na diff
  • Trudniejsze testy interaktywne (hover, focus states)
  • Brak wbudowanego HTML reportu z podglądem diffów

Podejście 2 — kod testu

e2e/visual/button.visual.spec.ts

import { test, expect, Page }
  from '@playwright/test';

async function gotoStory(
  page: Page,
  storyId: string
): Promise<void> {
  await page.goto(
    `/iframe.html?id=${storyId}&viewMode=story`
  );
  await page.waitForLoadState(
    'domcontentloaded'
  );
  await page.waitForSelector(
    'app-button', { state: 'visible' }
  );
  await page.waitForTimeout(300);
}

test.describe('Button – visual', () => {
  test.use({ viewport: {
    width: 400, height: 200
  }});

  test('Primary button', async ({ page }) => {
    await gotoStory(
      page, 'components-button--primary'
    );
    await expect(page)
      .toHaveScreenshot('button-primary.png');
  });
});

playwright.config.ts (fragment)

export default defineConfig({
  testDir: './e2e',
  use: {
    baseURL: 'http://localhost:6006',
    viewport: { width: 1280, height: 720 },
  },
  expect: {
    toHaveScreenshot: {
      maxDiffPixelRatio: 0.01,  // 1%
      threshold: 0.1,
    },
  },
  snapshotDir: '__snapshots__',
  webServer: {
    command: 'npm run storybook',
    url: 'http://localhost:6006',
    reuseExistingServer: true,
  },
});

GitHub Actions CI

# .github/workflows/chromatic.yml
name: Chromatic
on: [push]
jobs:
  chromatic:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # potrzebne do TurboSnap
      - run: npm ci
      - name: Run Chromatic
        uses: chromaui/action@latest
        with:
          projectToken: ${{ secrets.CHROMATIC_PROJECT_TOKEN }}
          exitZeroOnChanges: true  # nie blokuj CI, czekaj na review

Podejście 3 — zalety i wady

✅ Zalety

  • UI Review — reviewer zatwierdza zmiany wizualne w przeglądarce, jak code review ale dla wyglądu
  • Zero konfiguracji — automatycznie wykrywa wszystkie stories, bez pisania testów
  • Historia buildów — pełna historia snapshotów per branch i commit
  • Natywna integracja z GitHub/GitLab — PR check "UI Tests" blokuje merge do zatwierdzenia
  • TurboSnap — testuje tylko komponenty których pliki zmieniły się w PR (szybciej)
  • Renderowanie w chmurze — spójne wyniki niezależnie od OS dewelopera
  • Snapshoty trzymane w chmurze — nie zaśmiecają repozytorium

⚠️ Wady

  • Płatne powyżej 5000 snapshotów/miesiąc — przy dużym teamie i wielu PR może być kosztowne
  • Screenshoty i kod komponentów trafiają do zewnętrznego serwera (kwestia polityki bezpieczeństwa)
  • Wymaga dostępu do internetu — nie działa w pełni offline/air-gapped
  • Mniejsza kontrola nad środowiskiem renderowania vs lokalne narzędzia
  • Konto i token wymagane do startu — próg wejścia wyższy niż Playwright

Chromatic — czy można hostować we własnej infrastrukturze?

☁️ Chromatic Cloud (domyślny)

  • Kod i snapshoty trafiają na serwery Chromatic
  • Wymaga dostępu do internetu
  • Free do 5 000 snap/mies. · płatne powyżej
  • Gotowe natychmiast — zero DevOps

🏢 Chromatic On-Premises (Enterprise)

  • Pełne Chromatic na własnych serwerach
  • Kod i snapshoty nie opuszczają infrastruktury
  • Działa w środowiskach air-gapped (bez internetu)
  • Wymaga kontaktu z Chromatic i planu Enterprise

Alternatywy open-source (w pełni self-hosted)

Lost Pixel

Open-source, działa lokalnie lub na własnym CI. Podobny przepływ do Chromatic. GitHub: lost-pixel/lost-pixel

reg-suit + storycap

Storycap capture → reg-suit porównuje → raport HTML na własnym S3/GCS. Pełna kontrola, wymaga konfiguracji.

Playwright (już mamy)

Baseline w repo, raport HTML jako artefakt CI. W pełni lokalne, zero zewnętrznych serwisów.

Rekomendacja dla organizacji z restrykcyjną polityką bezpieczeństwa: Playwright (lokalnie) + Lost Pixel lub reg-suit (CI) — zero zależności od chmury zewnętrznej

  run: npm run test:visual:playwright

  uses: actions/upload-artifact@v4

Etap 3 — Skalowanie

Miesiąc 2+

  • Pokryć wszystkie komponenty design systemu
  • Dodać testy responsywności (mobile/tablet/desktop)
  • Testy dark mode / motywy
